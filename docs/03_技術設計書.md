- 作成日: 2026-07-06
- バージョン: 0.1（ドラフト）
- 関連文書: [01_企画書.md](01_企画書.md) / [02_プロダクト要求仕様書.md](02_プロダクト要求仕様書.md)

## 1. 設計方針

PRDの非機能要求から導かれる3原則をすべての設計判断の上位に置く。

1. **ファイルシステムが真実の源** — 原本は `書庫/docs/{到達番号}/` に無加工で平置き。SQLiteは索引に過ぎず、いつでも全損→再構築できる（NFR-01, 02）
2. **変換はすべてサーバー側・読み取り専用** — ブラウザには素のHTML/CSS/最小JSだけを返す。原本への書き込みは取り込み時の複製のみ（NFR-04）
3. **外に出ない** — `127.0.0.1` バインド、外部送信ゼロ、XSLT処理にもネットワークアクセスを許可しない（NFR-03）

## 2. 技術スタックと依存

| 領域 | 採用 | 備考 |
|---|---|---|
| ランタイム | Python 3.12+ / uv | `uv run kobunshoko` で起動（NFR-07） |
| Webフレームワーク | FastAPI + Uvicorn | SSR中心。テンプレートはJinja2 |
| XSLT変換 | lxml（libxslt） | XSLT 1.0完全対応。ブラウザ（旧Chrome/Safari）と同系エンジンで再現性最優先 |
| DB | 標準ライブラリ `sqlite3` + FTS5 | ORMは使わない（スキーマが小さく、再構築可能なため） |
| 文字コード判定 | charset-normalizer | CSV等のShift_JIS判定 |
| その他 | python-multipart（アップロード）、標準 `zipfile` / `hashlib` / `csv` | |

フロントはビルドなし。JSはドラッグ&ドロップと絞り込みフォーム送信程度に留める。

## 4. 設定と書庫レイアウト

設定の解決順: CLI引数 `--archive` → 環境変数 `KOBUNSHOKO_ARCHIVE` → 既定 `~/kobunshoko-archive/`。初回起動時にディレクトリを自動作成する。

```
~/kobunshoko-archive/
├── catalog.db                … SQLite（索引のみ）
└── docs/
    └── {到達番号}/
        ├── original.zip      … ZIP取り込み時のみ。元ZIPをそのまま複製
        └── （解凍された原本一式。サブフォルダ構造も維持）
```

- 到達番号はディレクトリ名としてそのまま使う（数字列なのでファイルシステム安全）
- 原本ファイル名は日本語のまま保持する（macOSはNFD正規化するため、DB照合時は `unicodedata.normalize("NFC", name)` で正規化して比較する）

## 5. データベース設計

```sql
CREATE TABLE meta (
  key TEXT PRIMARY KEY, value TEXT
);  -- schema_version など

CREATE TABLE documents (
  tno            TEXT PRIMARY KEY,   -- 到達番号
  received_date  TEXT,               -- ISO 8601（西暦正規化後）。抽出不能ならNULL
  received_raw   TEXT,               -- 和暦の原文（例: "令和 8年 6月27日"）
  agency         TEXT,               -- 発行機関
  title          TEXT,               -- 件名
  pattern        TEXT,               -- 適用した抽出パターン名。'fallback' = 形式未対応
  ingested_at    TEXT NOT NULL,      -- 取込日時（ISO 8601）
  verify_status  TEXT                -- NULL / ok / mismatch / unverifiable（P1）
);

CREATE TABLE files (
  id        INTEGER PRIMARY KEY,
  tno       TEXT NOT NULL REFERENCES documents(tno) ON DELETE CASCADE,
  name      TEXT NOT NULL,           -- NFC正規化済みファイル名（docs/{tno}/ からの相対パス）
  role      TEXT NOT NULL,           -- kagami / xsl / yoshiki / pdf / csv / zip / other
  sha256    TEXT NOT NULL,
  digest_ok INTEGER,                 -- NULL=未検証 1=一致 0=不一致（P1）
  UNIQUE (tno, name)
);

CREATE VIRTUAL TABLE search USING fts5(
  tno UNINDEXED, title, agency, body,   -- body = 鑑・様式XMLのテキストノード連結
  tokenize = 'trigram'                  -- 日本語の部分一致（FR-05）
);
```

- FTS5のトークナイザは `trigram` を採用（形態素辞書なしで日本語部分一致が成立する）。SQLite 3.34+ が必要 — Python 3.12同梱のSQLiteで満たせる
- **再インデックス**（NFR-02): `documents/files/search` を全削除し、`docs/*/` を走査して取り込み時と同じ抽出処理を再実行する。取り込みとコードパスを共有すること（挙動の乖離を防ぐ）

## 6. 取り込みパイプライン（FR-01, 02）

```
入力（ZIP or フォルダ）
  → [1] 展開・正規化 → [2] 鑑XML特定 → [3] メタデータ抽出
  → [4] 到達番号確定 → [5] 重複チェック → [6] 書庫へ複製 → [7] カタログ登録
```

失敗時は**どの段階でも書庫・DBに痕跡を残さない**: 一時ディレクトリ（`tempfile.mkdtemp`）で[1]〜[4]を行い、[5]通過後に `docs/{tno}/` へアトミックに移す（同一ボリューム内なら `os.rename`、書庫が別ボリューム指定の場合は copy→検証→rename）。DB登録[7]は最後に単一トランザクションで行う。

### 6-1. ZIP展開の注意点

- **ファイル名エンコーディング**: e-GovのZIPは日本語ファイル名を含む。`zipfile` はUTF-8フラグのないエントリ名をcp437として復号するため、フラグがない場合は `name.encode('cp437').decode('cp932')` で修復を試みる（失敗したらcp437のまま受け入れ、警告を記録）
- **Zip Slip対策**: 展開先パスを `resolve()` し、展開ルート配下であることを検証してから書き込む
- 入れ子ZIP（ZIP内ZIP）は展開しない。`role=other` としてそのまま保管
- フォルダ取り込みの場合は元フォルダを読み取り専用として扱い、複製のみ行う（FR-01: 副作用なし）

### 6-2. 鑑XML特定ヒューリスティック

ルート直下の `*.xml` を候補とし、次の順で確定する:

1. ルート要素が `DOC` で、`Signature`（XMLDSig）子要素を持つもの
2. `kagami.xml` という名前のもの
3. ファイル名が到達番号らしい数字列（14〜20桁）のもの
4. どれにも該当しなければ「鑑なし」として扱い、フォールバック登録（FR-02）

XMLパースは `lxml.etree.XMLParser(resolve_entities=False, no_network=True)` を用いる（XXE対策。公文書は信頼済みだが習慣として）。

### 6-3. 抽出パターンレジストリ

```python
class KagamiPattern(Protocol):
    name: str
    def matches(self, root: etree._Element) -> bool: ...
    def extract(self, root: etree._Element) -> Meta: ...
        # Meta = tno, received_raw, agency, title, doclinks

PATTERNS = [TotatunoPattern(), DocnoPattern()]  # 先勝ち。追加はここに1行
```

| パターン | 判定 | 抽出元 |
|---|---|---|
| `totatuno`（厚労省審査結果系） | `BODY/TOTATUNO` が存在 | 到達番号=`TOTATUNO`、受付日=`UKEDATE`、機関=`ATESK` または `SYOZOKUSOSIKI`、件名は固定「電子申請に対する審査結果について」（XSL上の見出しに一致） |
| `docno`（年金機構kagami系） | `BODY/DOCNO` が存在 | 到達番号=`DOCNO`、日付=`BODY/DATE`、機関=`AUTHOR/AFF` または `AUTHOR/NAME`、件名=`TITLE`、添付=`APPENDIX*/DOCLINK` |
| `fallback` | 常にtrue（最後） | 到達番号=フォルダ/ZIP名から数字列抽出（不能なら `unknown-{取込時刻}`）、日付=ファイル更新日時、件名=ZIP/フォルダ名。`pattern='fallback'` で形式未対応フラグ相当 |

### 6-4. 和暦→西暦変換

- 対応元号: 令和(2019)・平成(1989)・昭和(1926)。「元年」= 1年
- 全角数字・空白（`令和 8年 6月27日` の揺れ）を吸収してからパース
- 変換不能な場合は `received_date=NULL` とし `received_raw` のみ保持（一覧は取込日でソート可能なので致命的でない）

### 6-5. ファイルの役割判定（files.role）

拡張子と参照関係で決める: 鑑XML=`kagami` / `.xsl`=`xsl` / 鑑の `DOCLINK` から参照されるXML=`yoshiki` / `.pdf`=`pdf` / `.csv`=`csv` / 元ZIP=`zip` / それ以外=`other`。

## 7. 閲覧・変換パイプライン（FR-04）

### 7-1. XML → HTML（XSLT変換）

```
GET /doc/{tno}/view/{name}
  1. パス検証（後述 9-1）
  2. XML内 <?xml-stylesheet?> のhref → 同一文書セット内のXSLに解決
     （URLデコード必要: href は "%E9%91%91..." のようにエンコード済みの場合がある）
  3. lxml.XSLT で変換。XSLTAccessControl(read_network=False, write_network=False,
     read_file=True ※文書セットディレクトリ配下のみ, create_dir=False, write_file=False)
  4. str(result) でUnicodeとして取り出す（XSL側の shift_jis 出力指定を無視して
     UTF-8で配信。<meta charset> がHTML内にあれば除去または書き換え）
  5. テンプレートの iframe（sandbox属性、allow-scriptsなし）内に srcdoc で埋め込む
```

- **iframe sandbox を採用する理由**: XSLの出力は旧式HTML（IE時代のalign属性等）でありアプリのCSSと衝突させたくないこと、XSLにscriptが含まれる可能性を遮断すること（読み取り専用アーカイブの原則）。sandboxはスタイル適用を妨げないので表示品質に影響しない
- 旧式HTML対策のCSS正規化（`table { border-collapse: ... }` 等の最小セット）はsrcdoc冒頭に注入する
- XSLが見つからない/変換例外の場合は**汎用XMLツリー表示**（要素名と値をネストした`<dl>`で整形、`Signature`要素は折りたたみ）へフォールバックする

### 7-2. CSV表示

1. バイト列を読み、charset-normalizerで判定（cp932優先候補として与える）
2. `csv.reader` でパースし、テーブル表示。セル内の `<br/>` や `<a href>` 断片（年金機構CSVに実在）は**エスケープしてそのまま文字として表示**（HTMLとして解釈しない）
3. 判定失敗時は生バイトのhexダンプではなく「ダウンロードしてください」表示に切り替え

### 7-3. PDF・原本配信

- `GET /doc/{tno}/raw/{name}`: 原本をそのまま配信。PDFは `Content-Disposition: inline`（ブラウザ内蔵ビューア）、その他は `attachment`
- 日本語ファイル名は `filename*=UTF-8''...`（RFC 5987）で指定
- XML原本の`raw`は `text/plain` で返す（`text/xml` で返すとブラウザがXSLTを試みて挙動が環境依存になるため。原本バイト列はダウンロード時に完全一致）

## 9. セキュリティ設計

### 9-1. パス検証（全ファイル系エンドポイント共通）

```python
base = (archive / "docs" / tno).resolve()
target = (base / name).resolve()
if not target.is_relative_to(base): raise HTTPException(404)
```

`tno` は `[0-9A-Za-z_-]+` に制限。`name` はNFC正規化後にDBの `files.name` と照合し、**DBに登録されたファイルのみ**配信する（書庫内の未登録ファイルにもアクセスさせない）。

### 9-2. その他

- Uvicornは `host="127.0.0.1"` 固定。`--host` の外部公開オプションは設けない（案C移行時に別途設計）
- 外部リソース取得ゼロ: XSLTAccessControlでネットワーク遮断、XMLパーサは `no_network=True`、フロントもCDN参照なし
- 変換HTML内のリンク（年金機構の案内URL等）は `target="_blank" rel="noopener"`。iframe sandboxにより自動遷移はしない

## 10. 主要な設計判断（Decision Log）

| # | 判断 | 理由・却下した代替案 |
|---|---|---|
| D-01 | XSLT変換はキャッシュせず都度実行 | サンプル規模のXSLT 1.0変換はミリ秒オーダーでNFR-05を余裕で満たす見込み。キャッシュは無効化バグの温床。実測で問題が出た場合のみ `書庫/cache/` を導入（原本ディレクトリには置かない） |
| D-02 | ORM不採用・生SQL | テーブル3枚・再構築可能なDBにマイグレーション機構は過剰。`schema_version` で将来の変更に備える |
| D-03 | 変換HTMLはiframe(sandbox)+srcdoc | 旧式HTMLの隔離とscript遮断を同時に達成。shadow DOM案はJS必須になるため却下 |
| D-04 | XML原本のrawはtext/plain配信 | text/xml配信はブラウザのXSLT実装差（廃止進行中）で挙動が割れるため |
| D-05 | 実公文書をテストフィクスチャにしない | 個人情報・法人情報を含むためリポジトリにコミットせず、構造だけ模した合成フィクスチャを作る。実サンプルでの確認は手動の受け入れテストとする |
| D-06 | 到達番号をURLとディレクトリ名の主キーに使う | 全形式で一意に存在し、ファイルシステム・URL安全。フォールバック時のみ合成ID |

## 11. エラーハンドリング方針

- 取り込み: 文書セット単位でall-or-nothing。複数ZIP一括時は個別に成否を集計し、結果ページで「成功/スキップ(重複)/失敗(理由)」を列挙（FR-01, 画面要求）
- 閲覧: 変換失敗は汎用ツリー表示へフォールバックし、画面上部に「XSLT変換に失敗したため簡易表示しています（理由）」を表示。500は返さない
- ログ: 標準の `logging` でコンソールへ。ファイルログは持たない（書庫を汚さない）

## 12. テスト方針

- `tests/fixtures/` に2形式（totatuno系・docno系）の**合成文書セット**を用意: 実サンプルと同じ要素構造・エンコーディング（shift_jis CSV含む）でダミー値を入れたもの。ZIP版（cp437名修復ケース含む）とフォルダ版の両方
- 単体: 和暦変換、パターン抽出、ZIPファイル名修復、パス検証、CSV文字コード判定
- 結合: 取り込み→一覧→閲覧→再インデックスの一連をFastAPI TestClientで（受け入れ基準1, 5, 6, 7に対応）
- 受け入れ基準2〜4（実サンプルの表示同等性・実ブラウザ3種）は手動確認とし、確認手順を `docs/` に別途記録する
