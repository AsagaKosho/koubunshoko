# e-Gov電子申請APIによる公文書取得の調査

- 作成日: 2026-07-06
- ステータス: 調査完了・着手判断待ち
- 関連文書: [02_プロダクト要求仕様書.md](02_プロダクト要求仕様書.md)（スコープ外「e-Gov APIによる公文書の自動取得」の再検討材料）

## 調査の目的

公文書庫に「e-Govから公文書を直接取得するUI」を追加することが現実的かを判断する。

## 結論

**技術的には現実的（公式APIが存在する）。ボトルネックはコードではなく利用登録・試験のプロセス。**
検証環境は即日使えるため、プロトタイプ→本番化の二段構えを推奨する。

## 調査結果

### 公式API の現状

- e-Govには公式の**電子申請API**があり、申請提出・処理状況照会・**公文書取得（ZIP）**が可能。労務SaaS（SmartHR・freee・ジョブカン等）の公文書自動取得はこれを利用している
- 旧・**外部連携APIは2024年1月31日でサポート終了**。今から実装するなら電子申請API一択
- 参考実装として SmartHR がOSSライブラリ kiji（Ruby、旧API向け）を公開していた実績がある

### 利用までの段取り

| 段階 | 内容 | ハードル |
|---|---|---|
| 1. 検証環境 | 開発者ポータルでアカウント登録（仮登録→本登録）。検証環境のAPIキーは**即時発行で使える** | 低い（無料・即日） |
| 2. 本番環境 | 「最終確認試験」の申込を経て本番用ソフトウェアID・APIキーの発行 | 申込書ベースの審査プロセス。建付けはソフトウェア開発事業者向け |

### 技術面

- APIキーによるクライアント認証＋利用者のe-Govアカウント認可
- 公文書一覧の取得 → 公文書ZIPのダウンロード、という流れ
- 取得したZIPは公文書庫の既存取り込みパイプライン（`ingest_zip`）にそのまま流せるため、**公文書庫側の追加実装は数日レベル**と見積もる
- UI案: 「未取得の公文書を定期チェックして自動取り込み」または「到達番号を指定して取得」

### 注意点・制約

- 取得できるのは**自分のアカウントに紐づく申請案件の公文書**が基本
- 公文書庫に初めて外部通信（e-Gov本体へのHTTPS）が入るため、実装時はNFR-03（外部送信ゼロ）に**例外を明記**する設計変更が必要。認可トークンのローカル保持も新たな考慮点
- マイページのブラウザ自動操作（スクレイピング）による代替は、ログイン・2段階認証・画面変更に脆く利用規約リスクもあるため**不採用**

### 未確認事項（検証環境で要確認）

1. **紐付けの適用範囲**: Web画面から申請した案件・社労士ソフト経由の案件の公文書をAPIで取得できるか。旧APIには「申請案件の紐付け設定」があったが、新APIでの仕様は一次情報にあたれていない
2. **本番申込の主体**: 個人（または小規模法人の自社利用）で最終確認試験〜本番発行まで通るか
3. 年金機構「オンライン事業所年金情報サービス」経由の電子送付文書（被保険者データ等）がAPIの取得対象に含まれるか

## 推奨する進め方

1. **現状維持でも実用十分**: 監視フォルダ機能（`--watch-dir ~/Downloads`）で「ブラウザでダウンロードボタンを押すだけ」の運用がすでに可能（取得UIの体験の8割）
2. それでも欲しければ **検証環境でプロトタイプ**を作り、上記の未確認事項1〜3を実地で確認
3. 実用に耐えると判断できたら最終確認試験を申し込み本番化。その際にPRD・技術設計書（NFR-03例外、トークン管理、`fetch` モジュール追加）を改訂
