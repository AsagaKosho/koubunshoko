# e-Gov 公文書アーカイブ「公文書庫」 調査・企画書

- 作成日: 2026-07-06
- ステータス: 承認済み（決定事項は末尾参照）
- 関連文書: [02_プロダクト要求仕様書.md](02_プロダクト要求仕様書.md)

## 要旨

ZIPを放り込むだけで公文書を整理・検索でき、任意のブラウザで閲覧できる**ローカルWebアプリ**を作る。XSLT変換はブラウザに任せず**サーバー側で実行**する。

なぜ今やるか:

1. Chromeが2026年11月にXSLTを廃止し、「XMLをブラウザで直接開く」従来の方法が使えなくなる
2. e-Govの公文書は発行後3か月で取得期限が切れるため、手元での恒久アーカイブが必須

---

## 第1部 調査

### 1-1. サンプル公文書の分析

手元の2件のサンプルを解析した。発行元が違うとXMLスキーマもファイル構成も異なるが、共通の骨格がある。

| | 労働基準監督署（審査結果通知） | 日本年金機構（納入告知額通知） |
|---|---|---|
| 鑑XML | `DOC/BODY` 直下に `TOTATUNO`（到達番号）、受付日、法人名、宛先署など | `DOC/FRONT+BODY`。`DOCNO`、発行者、`APPENDIX/DOCLINK` で添付を列挙 |
| 表示用XSL | 鑑文書表示用スタイルシート.xsl（XSLT 1.0） | `kagami.xsl` ＋ 様式本体用の `yoshiki_29_zumitsu_001.xsl` |
| 添付 | PDF（控え文書） | 様式XML（通知書本体）＋ CSV明細（Shift_JIS） |
| 署名 | XMLDSig（RSA-SHA256、政府認証基盤GPKIの証明書）。添付ファイルごとのSHA-256ダイジェスト付き | 同左 |

共通の骨格:

```
到達番号のZIP（例: 209906019912345678_20260706120000.zip）
└─ 到達番号のフォルダ/
   ├─ 鑑XML …… メタデータ＋XMLDSig署名（システムの「入口」になる）
   ├─ 表示用XSL …… 鑑・様式をHTMLに変換（XSLT 1.0）
   └─ 添付 …… PDF / 様式XML / CSV など。鑑XMLから参照される
```

- XMLは `<?xml-stylesheet?>` で必ず対のXSLを参照する。**XMLとXSLを分離すると表示できなくなる**ため、フォルダ単位での保全が前提
- XSLの出力エンコーディングが `shift_jis` 指定、CSVもShift_JIS。UTF-8前提の現代ブラウザ・ツールでは文字化けの温床
- 署名の `Reference` に各添付のSHA-256が入っており、**改ざん検知（原本性確認）がローカルで可能**

### 1-2. ブラウザのXSLT廃止 — 従来手法の寿命

これまで「XMLをダブルクリック → ブラウザがXSLで変換して表示」が公式の閲覧方法だったが、Chromeはセキュリティ上の理由（libxsltの脆弱性）でXSLTの廃止を決定済み。FirefoxとSafari（WebKit）も追随を表明している。

| 時期 | イベント |
|---|---|
| 2025-12（Chrome 143） | 非推奨警告の表示開始 |
| 2026-03（Chrome 146） | 企業ポリシーでの無効化テスト開始 |
| **2026-11-17（Chrome 158）** | **安定版でXSLT停止 — 一般ユーザーはXML直接表示が不可能に** |
| 2027-08（Chrome 176） | 企業ポリシーの猶予も終了 |

> **あと4か月ほどで従来の閲覧方法が壊れる。** なお現時点でも `file://` で開いたXMLへのXSLT適用はセキュリティ制限でブロックされるため、実質的にすでに「ダブルクリックで開けない」状態。この空白を埋めることが本システムの中核価値になる。

### 1-3. e-Gov側の制約

- 公文書は**発行後3か月で取得期限切れ**。e-Gov上には残らないので、ダウンロードしたZIPが唯一の原本。散逸・削除リスクが高い
- ZIPと解凍済みフォルダが混在し、フォルダ名が到達番号（数字の羅列）のため**内容が名前から判別できない**
- 社労士向け市販ソフトに「公文書表示」機能はあるが、恒久アーカイブ・横断検索・任意ブラウザ閲覧を満たす軽量な自前ソリューションは見当たらない

## 第2部 企画

### 2-1. コンセプト

> 「e-GovからダウンロードしたZIPをそのまま放り込むと、発行機関・日付・件名で整理された書庫になり、どのブラウザからでも読める。」

原本（ZIP・XML・署名）には一切手を加えず保全し、閲覧用の変換はすべてその場で行う。ブラウザ側に何も要求しないので、XSLT廃止後も、他のPCからでも使える。

### 2-2. 機能（MVP）

1. **取り込み** — ZIPまたはフォルダをドラッグ&ドロップ（または監視フォルダから自動取り込み）。解凍し、鑑XMLから到達番号・受付日・発行機関・件名・添付一覧を抽出してカタログ化。原本ZIPも保管
2. **一覧・検索** — 発行機関、年月、種別、フリーワードで絞り込み
3. **閲覧** — XML＋XSLを**サーバー側でXSLT 1.0変換**してHTML配信（Shift_JIS指定はUTF-8に正規化）。PDFはそのまま、CSVは文字コード変換のうえテーブル表示
4. **原本性の担保** — XMLDSigの `Reference` ダイジェストと実ファイルのSHA-256を照合し、「改ざんなし」バッジを表示

Phase 2以降: 署名の証明書チェーン検証（GPKI）、タグ・メモ、監視フォルダ、バックアップ連携。

### 2-3. アーキテクチャの選択肢

| 案 | 概要 | 評価 |
|---|---|---|
| **案B: ローカルWebアプリ（採用）** | 手元で小さなサーバーを起動し `http://localhost:8xxx` をブラウザで開く。変換・検索・保管をすべてサーバー側で処理 | どのブラウザでも動く（XSLT廃止の影響ゼロ）。ファイル管理・全文検索が自然に実装できる。LAN内共有への拡張も容易。欠点は起動の一手間（自動起動設定で解消可能） |
| 案A: サーバーレスの静的SPA | ブラウザ内でJS製XSLTエンジン（SaxonJSなど）を使い変換 | 配布は楽だが、ローカルファイルアクセスがFile System Access API（Chrome系限定）頼みになり「任意のブラウザ」の要件と衝突。管理機能も弱い |
| 案C: クラウド常設 | 複数人・複数端末での共有に最適 | 機微情報（保険料額・従業員情報）を外部に置く判断が必要。必要になったら同じコードをVPN/Tailscale内に置く発展形とする |

### 2-4. 推奨技術スタック

| 領域 | 選定 | 理由 |
|---|---|---|
| XSLT変換 | Python `lxml`（libxslt） | ブラウザが今まで使ってきたのと同じ系統のエンジンで、XSLT 1.0を完全サポート。表示の再現性が最も高い |
| サーバー | FastAPI ＋ Uvicorn | `uv run` 一発で起動できる構成に |
| カタログ | SQLite（FTS5で日本語全文検索） | 単一ファイルでバックアップも容易 |
| フロント | SSR中心の素朴なHTML＋最小限のJS | ビルド不要 |
| 保管構造 | `書庫ディレクトリ/到達番号/` に原本一式＋元ZIP | DBは索引のみ持ち、原本は常にファイルシステムに平置き（システムが死んでもファイルは残る） |

### 2-5. データモデル（素案）

```
documents   … 到達番号(主キー), 受付日, 発行機関, 件名, 種別, 取込日, 署名検証結果
files       … 文書ID, ファイル名, 役割(鑑/XSL/様式/PDF/CSV), SHA-256, ダイジェスト照合結果
search      … FTS5仮想テーブル（件名・機関・XML本文テキスト）
```

未知のスキーマ（他省庁の様式）が来ても、「XML＋対のXSLをそのまま変換して表示」という省庁非依存の方式なので閲覧は常に成立する。メタデータ抽出だけ、鑑XMLのパターン（`TOTATUNO` 系・`DOCNO` 系…）を順に試し、全部外れたらファイル名と日付でフォールバックする。

### 2-6. リスクと対策

| リスク | 対策 |
|---|---|
| 省庁ごとにXMLスキーマがバラバラ | 表示はXSLT任せで吸収。メタデータ抽出はパターン追加式にし、未知形式も「生で保管＋汎用表示」で受ける |
| Shift_JIS混在による文字化け | 取り込み時に判定し、閲覧時は常にUTF-8へ正規化。原本は無変換で保持 |
| XSLが古いHTML（IE時代の属性）を出力 | 実害は軽微。表示コンテナ側に軽いCSS正規化を当てる |
| 署名検証の本格実装（GPKI証明書チェーン）が重い | MVPはダイジェスト照合（改ざん検知）まで。チェーン検証はPhase 2以降 |

### 2-7. ロードマップ

| フェーズ | 内容 |
|---|---|
| Phase 1 | 取り込み（ZIP/フォルダ）＋一覧＋XSLT閲覧＋PDF/CSV表示。これだけで「開けない・見つからない」は解消 |
| Phase 2 | 全文検索（FTS5）、ダイジェスト照合バッジ、監視フォルダ自動取り込み |
| Phase 3 | GPKI署名チェーン検証、タグ・メモ、LAN内共有、バックアップ連携 |

## 決定事項（2026-07-06）

- **利用者の範囲**: 自分専用（単一ユーザー）。認証・共有機能はスコープ外
- **書庫の置き場所**: ローカルディスク
- **未決**: Documents内に散在する過去分ZIPの一括取り込みの範囲
